# KBase 私有化智能知识库 —— 技术方案汇报

- 汇报对象：新疆生产建设兵团相关业务与信息化部门
- 汇报日期：2026-07-08（周三）
- 文档性质：技术测试演示配套材料，会后留存 / 领导传阅

---

## 一、背景与需求理解

兵团在政策管理与内部行政场景中，存在三类具体的知识密集型工作：

1. **政策方案生成**：围绕某一主题（如人才引进、住房保障），需要先搭出方案大纲，再逐节查阅政策依据、撰写正文，最终形成一份可引用、可溯源的成文方案。
2. **政策摘编与定期汇编**：面对持续增长的政策文件库，需要定期产出多文档摘要汇编，供领导快速掌握文件全貌。
3. **财务报销多轮问答**：报销经办与员工咨询报销标准时，问题往往是连续追问式的（例如先问"出差住宿费标准是多少"，接着追问"那司局级呢？"），要求系统能理解上下文、不丢失指代关系。

这三类场景的共同难点不是"要不要用大模型"，而是：

- 答案必须**有据可查**，不能靠模型自由发挥（涉政务合规与责任追溯）；
- 文档中大量**文号、人名、专有名词等精确表述**，普通语义检索容易漏召回；
- 数据**不能出域**，尤其涉及内部政策文件时；
- 现场使用者对"AI 会不会编造"心存疑虑，需要机制层面的防呆设计，而非仅靠模型自觉。

KBase 是围绕这四点难点设计的私有化知识库系统，本次技术演示将逐一对应展示。

## 二、总体方案与架构

KBase 采用分层可插拔架构，内核只依赖抽象接口，具体实现（Embedder / 向量库 / LLM / OCR / 分块器）在插件层注册、由 YAML 配置选择，同一套内核代码可以从单机演示形态平滑升级到集群生产形态。

```
┌──────────────────────────────────────────────────┐
│ 接入层   Web 管理与问答界面 / REST API / MCP Server  │
├──────────────────────────────────────────────────┤
│ 应用层   逐轮问答(RAG) / 定期汇编 / 方案生成(内置编排) │
├──────────────────────────────────────────────────┤
│ 内核层   摄取管道 → 分块 → 索引 → 混合检索(+重排) → 生成│
├──────────────────────────────────────────────────┤
│ 插件层   Embedder / VectorStore / LLMProvider /    │
│          OCRBackend / Chunker / Enricher（均可替换）│
├──────────────────────────────────────────────────┤
│ 存储层   向量库 / 元数据库 / 原始文件+Markdown 双存   │
└──────────────────────────────────────────────────┘
```

核心设计取舍：

- **不绑定任何第三方 Agent/RAG 框架本体**，自研接口层 + 精选原子库，理由是私有化涉密交付要求依赖可审计、许可干净，且内置编排（方案生成/汇编）需要长在自有内核之上；
- **文档双存**：原始文件与转换后的 Markdown 都保留，重建索引或更换分块策略无需重新解析原始文件；
- **模型可切换**：LLM Provider 在配置中声明、界面下拉即可切换，当前已验证 qwen-plus / qwen-max / deepseek-v3 / qwen3-32b，私有化部署时可替换为客户自有的本地大模型（同一套 OpenAI 兼容接口协议）。

## 三、三大场景：需求 → 能力 → 演示点

### 场景一：政策方案生成

**需求**：给定主题（如"人才引进住房保障实施方案"），产出结构完整、逐节有据、可导出的正式方案文档。

**能力**：

1. 系统先用一次 LLM 调用生成 3~7 节的大纲（结合检索到的相关政策背景），大纲可在界面编辑（增删节、改标题、调顺序）；
2. 确认大纲后转入后台任务：**每一节独立检索**（主题 + 节标题 + 要点，取相关度最高的若干片段）并生成正文；某节确实检索不到依据时，系统不会让模型编造，而是直接写入"知识库中无相关依据，本节未生成"的占位说明，且不影响其余节继续生成；
3. 全部节完成后自动汇整：按"文档名+标题路径"对所有节的引用做全局去重编号，正文中的引用角标重新映射为统一编号，文末生成"引用文献"附录；
4. 产物支持 Markdown 与 Word（docx）两种格式导出。

**实测数据**：7 节方案实测总耗时 **439 秒**，全部 7 节生成成功（无失败节）。

**演示点**：现场发起一次生成任务后，切换到提前准备好的已完成任务，展示完整产物结构、引用编号、docx 导出效果——439 秒的生成过程不适合现场干等，因此现场只做"发起"动作，产物展示走预生成的成品任务（详见《演示脚本》风险预案）。

### 场景二：政策摘编 / 定期汇编

**需求**：对知识库内多份文档定期产出摘要合集，便于领导快速通览。

**能力**：

- 可指定文档集合，也可省略后默认汇总该知识库内全部已就绪文档；
- 每份文档独立生成一段 200 字以内摘要，全部完成后再生成一段总览；
- 产物固定结构：标题 + 总览 + 逐文档摘要段落，同样支持 Markdown / docx 导出；
- 与方案生成共用同一套后台任务/进度轮询机制，单文档摘要失败不影响其余文档。

**演示点**：知识库页选定若干已摄取文档，发起汇编任务，展示进度条与最终汇编产物结构。

### 场景三：财务报销多轮问答

**需求**：报销经办与员工围绕报销标准连续追问，系统需理解追问中的省略与指代。

**能力**：

- 多轮会话中，像"那司局级呢？"这类依赖上一轮上下文的追问，字面检索往往因关键词不完整而拒答；
- 系统在检索前用 LLM 把这类追问改写为语义完整、可独立检索的问题（QueryRewrite），仅影响检索这一步——生成回答与会话记录仍固定使用用户原文，界面上看到的问题和历史不受影响；
- 触发是启发式的（问题较短、含指代词、或与历史几乎无重叠才触发改写），自包含的完整问题不会被误改写，避免"过度联想"；
- 改写失败（LLM 超时/报错/空输出）静默回退为原问题，不阻塞主查询链路。

**演示点**：先问"出差住宿费标准是多少？"，追问"那司局级呢？"，展示第二问依然准确命中并给出引用，同时打开引用抽屉回跳原文验证出处。

## 四、引用溯源与防幻觉：核心可信机制

三大场景共享同一套"防止 AI 胡说"的底层机制，这是本产品面向政务/涉密场景的核心差异化设计：

1. **全链路引用溯源**：每一句生成的答案都可以点击引用角标 `[n]`，弹出引用抽屉查看命中片段、相关度分数，并可跳转到原文文档全文定位处，让"AI 说的话"始终可被人工核实。
2. **拒答门（防幻觉）**：当检索到的证据相关度低于阈值时，系统明确回答"知识库中未找到依据"，而不是让模型编造内容；拒答被视为正常响应路径而非错误，并在日志中单独归类，可作为知识库覆盖度的运营指标。
3. **冲突证据保留**：当检索到的多个片段之间存在信息冲突时，系统不做静默取舍，而是保留冲突证据一并提供给模型辨析，并在回答中提示这一情况，交由使用者进一步判断，避免"看似确定实则各执一词"的误导。
4. **混合检索保证精确词不丢**：政策文档中大量文号、人名、机构名等专有名词，纯语义向量检索容易漏召回；KBase 采用向量语义检索 + 关键词检索双路并行，通过 RRF（倒数排名融合）算法合并，再经过重排模型精排，兼顾语义理解与精确匹配。

## 五、扫描件处理（OCR）

政策文件中普遍存在扫描件/图片版文档。KBase 摄取管道会自动探测文档是否有文本层：

- 有文本层的 PDF/Office 文档直接解析；
- 扫描件/图片文档路由至 OCR 服务（MonkeyOCR，GPU 部署），转换为标准 Markdown 后进入同一套分块/索引流程；
- OCR 服务不可用时，文档优雅降级为"待 OCR"状态而非失败，不阻塞同批次其余文档摄取，服务恢复后可批量重试。

**实测数据**：扫描件 PNG 样本识别耗时 **13 秒**，置信度 **1.0**。低置信度文档会在界面标黄提示人工核对。

## 六、安全合规

面向私有化政务交付，KBase 内置以下安全机制：

| 机制 | 说明 |
|---|---|
| 三级角色 | admin / editor / viewer 严格序权限，路由级校验；viewer 只能查看与问答，editor 可管理知识库与发起生成任务，admin 独占 Provider/用户/API Key/审计管理 |
| 审计日志 | 记录全部变更类请求、登录成败、问答行为（含问题摘要），管理员可查询 |
| API Key | 支持创建/吊销，完整密钥仅创建时展示一次，数据库只存哈希，供外部系统/MCP 集成使用 |
| 会话安全 | JWT 会话签名密钥可显式配置，避免重新部署后所有会话失效；生产要求反向代理终结 TLS |
| 许可证 | 轻量许可证校验（机构名/有效期/签名），v1 不锁功能，仅作状态展示 |
| 数据不出域 | 全私有化部署，向量库、元数据库、大模型均可部署在客户内网，无需依赖外部云服务 |

## 七、性能实测与硬件配置建议

### 7.1 实测结论（诚实汇报，用于硬件预算测算依据）

测试环境：4 vCPU + 单张 NVIDIA L4 GPU（与 OCR 服务共享同一张卡）。

| 并发档位 | 检索 P95 延迟 | 说明 |
|---|---|---|
| 10 并发 | **0.86 秒** | 全部走完整精排链路（向量+关键词+融合+重排） |
| 100 并发 | **4.2 秒** | 63% 的请求自动降级为"混合检索（不重排）"，降级请求仍带引用、非空结果，降级比例可通过监控指标观测 |

关键结论：

- **瓶颈是重排环节的 GPU 计算吞吐，不是显存容量**——实测显存占用远未打满，真正受限的是重排模型（交叉编码器）单次推理的物理耗时，与批大小/并发数无关；
- 系统内置**自适应降级**机制：当重排请求量超出配置的并发上限时，不会让查询排队等待 GPU，而是直接降级为"融合排序但跳过重排"，牺牲部分精排质量换取响应速度，且降级状态可通过 `/healthz` 与检索调试接口全程观测，不是静默劣化；
- 已验证"调大线程池/连接池"等常规调优手段在当前 4 vCPU 机型上收效有限甚至轻微回退，真正的扩容方向是**独立的重排算力（GPU）**。

### 7.2 硬件配置分档建议

| 使用场景 | 建议配置 | 说明 |
|---|---|---|
| 演示 / 小组级验证 | 单机 CPU（lite 部署形态） | 即开即用，无需额外硬件，适合科室级试用与效果验证 |
| 部门级（≤30 并发） | 单张 L4 / RTX 4090 级 GPU | 满足部门日常问答与生成场景的并发需求 |
| 全员级（100 并发，全精排 P95 < 500ms） | 独立重排 GPU 一张 + 主检索/生成卡一张 | 重排与生成算力分离，避免相互挤占；具体型号可根据实际并发预算进一步测算 |

> 说明：以上分档基于当前实测数据外推的工程建议，具体采购前建议在客户目标并发量级上做一轮现场压测复核。

## 八、质量保障

- **自动化测试**：后端 333 项、前端 76 项自动化测试，覆盖插件契约、检索链路、权限矩阵、生成任务、前端组件逻辑等。
- **检索质量评测**：内置评测工具对比"纯向量 / 混合检索 / 混合检索+重排"三档命中率，当前评测集下三档均达到 **5/5** 全部命中，验证了混合检索与重排两个增强环节确有增益、至少不劣化。
- **真实环境全栈验证**：在 GCP L4 实例上完整部署 standard 生产形态（PostgreSQL + Qdrant + TEI 向量/重排服务），全链路功能与压测均基于真实部署验证，非纸面设计。

## 九、部署形态对照

KBase 提供两种部署形态，共享同一套应用代码，仅通过配置切换底层组件实现：

| 对比项 | lite（演示/小规模） | standard（生产/较高并发） |
|---|---|---|
| 元数据库 | SQLite（单文件） | PostgreSQL |
| 向量库 | Chroma（嵌入式，进程内） | Qdrant（独立服务） |
| 向量化/重排 | 进程内加载模型 | TEI 独立推理服务（可 GPU 加速） |
| 容器数量 | 1 个 | 5 个（应用 + PG + Qdrant + 2×TEI） |
| 适用场景 | 快速验证、单机演示、小规模试用 | 部门级以上、需要横向扩展的生产部署 |
| 一键部署 | `docker compose -f docker-compose.lite.yml up` | `docker compose -f docker-compose.standard.yml up` |

两种形态间目前是"原件重摄取"式手工迁移（原始文件保留，重新上传到新栈即可），暂无自动化迁移工具，已在文档中如实说明。

## 十、后续路线图与合作方式

当前已完成的能力覆盖了三大场景的核心闭环（检索、生成、多轮问答、安全权限、私有化部署）。可选的下一阶段方向包括：

- **SSO / LDAP 对接**：与兵团现有身份认证体系集成，免去独立账号体系；
- **语义缓存**：政策问答重复率高的场景下，用语义缓存进一步提速、降低模型调用成本；
- **公文模板系统**：方案生成/汇编产物按兵团公文规范定制模板；
- **评测集共建扩充**：与业务方共建更大规模的检索评测集，持续校准检索与拒答阈值质量。

以上均为可选方向，具体优先级与合作方式（现场部署 / 远程支持 / 联合调优）可在本次技术演示后进一步商议。
