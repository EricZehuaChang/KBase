## 1. 备份与恢复

### 1.1 lite 档（一条命令）

所有状态都在 `data_dir`（默认 `./data`）：SQLite 主库、Chroma 向量、
原始文件、上传暂存。仓库自带工具整包备份：

```bash
# 备份（服务可不停——SQLite 走在线快照 API 保证一致性）
python scripts/backup.py backup --config config/kbase.yaml --out backups

# 恢复（必须先停服！）
python scripts/backup.py restore --archive backups/kbase-20260715-093000.tar.gz \
    --config config/kbase.yaml
```

要点：
- 备份产物：`backups/kbase-<时间戳>.tar.gz`，异地留存至少 7 份（每日一备）。
- 恢复不会直接删现场：原 `data_dir` 自动挪为 `data.pre-restore-<时间戳>`，
  确认恢复无误后再手动删除；恢复出错直接把它挪回来即回滚。
- **大批量导入进行中不要备份**（向量库与主库可能瞬时不一致），等导入完成。

定时备份（Linux crontab 示例，每天 02:30）：

```cron
30 2 * * * cd /opt/kbase && /opt/kbase/.venv/bin/python scripts/backup.py backup --config config/kbase.yaml --out /backup/kbase >> /var/log/kbase-backup.log 2>&1
```

### 1.2 standard 档（三件各备）

| 数据 | 工具 | 命令 |
|---|---|---|
| PostgreSQL 主库 | pg_dump | `pg_dump -Fc -h <host> -U kbase kbase > kbase-$(date +%F).dump` |
| Qdrant 向量 | 快照 API | `curl -X POST http://<qdrant>:6333/collections/<name>/snapshots`（逐 collection） |
| 原始文件目录 | scripts/backup.py | 同 lite（此时 data_dir 下无 sqlite，脚本自动跳过并提示） |

恢复顺序：**先 PG，再 Qdrant，最后文件目录**，全部完成后再启动服务。

```bash
pg_restore -d kbase --clean --if-exists kbase-2026-07-15.dump
# Qdrant: 把快照文件放回 snapshots 目录后
curl -X PUT http://<qdrant>:6333/collections/<name>/snapshots/recover \
     -H 'Content-Type: application/json' \
     -d '{"location": "file:///qdrant/snapshots/<name>/<snapshot-file>"}'
```

极端情形（向量库丢失但 PG 完好）：向量可以**从主库全量重建**——
chunks 表存有全部文本，逐库重嵌入即可（时长参考容量参数表：
TEI GPU 档约 50-100 块/秒）。主库才是唯一不可再生资产，备份优先级最高。

## 2. 版本升级

KBase 的 DB 迁移是**幂等自动**的（启动时补列/建表，无 Alembic、无手工
SQL）。升级核心流程五步：

```text
1. 备份            —— 见上文，升级前必做，这是唯一的回滚手段
2. 停服            —— systemctl stop kbase / docker compose down
3. 换代码          —— git pull（或解压新版本包）；依赖有变则 pip install -e .
4. 起服            —— 启动时自动跑迁移（日志可见 run_migrations）
5. 冒烟            —— 见下方清单，1 分钟
```

冒烟清单（升级后立即执行）：

```bash
curl -s http://localhost:8000/healthz          # status: ok，embedder/vectorstore 正常
curl -s http://localhost:8000/metrics | head   # 指标出口有输出
# 登录管理端：知识库列表可见 → 任选一库问一个已知答案的问题 → 引用正常
```

回滚：停服 → `git checkout <旧版本tag>` → `scripts/backup.py restore` 恢复
升级前备份 → 起服。**先回代码再回数据**（新版本可能已写入新列，旧代码
读新库通常兼容——迁移只加列不改列，但反向不保证，故数据一并回滚最稳）。

注意事项：
- Docker 部署升级 = 换镜像 tag 后 `docker compose up -d`，数据卷不动。
- **不要跳大版本**（如 M4 直升 M6）：迁移逐版本验证过，跨版本先在
  测试环境用备份副本演练。
- bcrypt 依赖锁定 `<4.1`，升级依赖时勿动（登录会全挂）。

## 3. 监控接入

- `GET /healthz`：存活探针（K8s liveness / 负载均衡健康检查用），
  返回 embedder/向量库/重排器状态。
- `GET /metrics`：Prometheus 文本格式（问答量、拒答数、重排降级等），
  无鉴权——靠网络隔离保护，勿暴露公网。抓取配置：

```yaml
scrape_configs:
  - job_name: kbase
    static_configs: [{ targets: ["kbase-host:8000"] }]
```

建议告警规则：`healthz` 连续 3 次失败；拒答率 5 分钟滑窗 > 30%
（知识缺口或检索故障信号，配合管理端"运营看板→无答案问题清单"排查）。
