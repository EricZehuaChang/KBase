# KBase 国际化(i18n)多语言支持 — 设计

> 2026-07-22 | 客户紧急需求(马来西亚:英语 + 马来语) | 分 P1/P2 两阶段交付

## 1. 目标

给 KBase 两个前端 SPA(使用端 portal / 管理端 admin)+ 后端错误提示增加多语言**界面壳**能力。首批语言:中文(源)、英语、马来语;架构支持后期快速接入新语言,且**译文可由运营在管理端后台编辑、即时生效**(机器翻译只是基线,意思一定可调)。

## 2. 范围

**做**:界面壳文案(菜单/按钮/提示/占位符/表头)+ 后端返回给用户的错误消息(105 处)。使用端 + 管理端全覆盖。

**不做**:用户上传的知识库文档内容、检索召回片段、LLM 问答回答——一律**保持原文档语言不翻译**(用户明确)。

## 3. 已确认决策

| 项 | 决策 |
|---|---|
| 架构 | **方案 A:代码基线 JSON + DB 覆盖层,运行时合并**(基线随版本走+离线兜底;DB 只存运营改过的增量;即时生效) |
| 语言集 | 中(zh,源)/ 英(en)/ 马来(ms) |
| 默认 + fallback | **中文**(现成全量,保证任何 key 都有值兜底) |
| 语言选择 | 浏览器自动检测 → 顶栏手动切 → localStorage 记住;登录用户可选存账号(P2) |
| i18n 引擎 | **vue-i18n**(MIT,复用其 fallback/插值/懒加载,不自研) |
| 机翻 | **DeepSeek**(config 已有 provider `deepseek-v4-flash`,零新依赖)批量把 zh → en/ms 生成基线 |
| key 设计 | 语义 key(`域.用途`,如 `kb.create`),不用中文当 key |
| 后端错误 | HTTPException detail → `{code, params, message}`;前端按 code 查 i18n 渲染,查不到用 message(中文原文)兜底 |
| 快速接新语言 | 加 `locales/xx.json`(DeepSeek 机翻生成)+ 语言清单加一项;DB 覆盖表天然支持,零结构改动 |

**不采用**完整 TMS 平台(Tolgee/Weblate/Traduora):它们是独立重服务(Java/Python/Node + Postgres),违背 KBase 私有化单容器定位;Weblate(GPL)/Traduora(AGPL)copyleft 对闭源交付有法律风险。只复用轻量现成件:vue-i18n(引擎)+ DeepSeek(机翻)+ 抽取工具(开发期)。

## 4. 数据结构

**基线文件**(随代码,机翻+校准后的底):
```
web-app/src/i18n/locales/zh.json   ← 源 + 默认 + fallback
web-app/src/i18n/locales/en.json
web-app/src/i18n/locales/ms.json
web-app/src/i18n/languages.ts      ← 支持语言清单 [{code,name,flag}];加新语言改这里
```
语义 key,按域分组,带插值:
```json
{ "common": { "save": "保存", "cancel": "取消" },
  "kb": { "create": "新建知识库", "empty": "还没有文档" },
  "error": { "kb_not_found": "知识库不存在: {id}" } }
```

**DB 覆盖表**(只存运营改过的 key):
```
translations(lang, key, value, updated_at, updated_by)   PK=(lang, key)
```

## 5. 数据流

**前端加载**:①定语言(localStorage → navigator.language 检测 → 默认 zh)②用打包基线 JSON **立即渲染**(无白屏)③异步 `GET /api/i18n/{lang}` 拉 DB 覆盖,`mergeLocaleMessage` 合并(DB 优先)。

**运营编辑**:管理端「多语言」页改 key → `PUT /api/i18n` 写 translations 表 → 当前页热更,其他会话下次加载生效。

## 6. 组件设计

### 前端
- `web-app/src/i18n/index.ts` — `createI18n()` 工厂(vue-i18n,locale=检测结果,fallbackLocale=zh,messages=基线)+ `loadDbOverrides(lang)`(拉 API 合并)+ `setLanguage(lang)`(切换+localStorage+可选账号)。两个 SPA 各自 `main.ts` `.use(i18n)`(bundle 隔离,各拉一次覆盖)。
- `web-app/src/components/LanguagePicker.vue` — 顶栏地球图标下拉,portal + admin 复用。
- 错误渲染:`lib/api/core.ts` 拦截器读 `detail.code` → `t(code, params)`,无 code/查不到 → `detail.message` 或原字符串。

### 后端
- `kbase/models.py` — `Translation` 表(create_all 自动建,lite/standard 同)。
- `kbase/i18n_store.py` — 覆盖表 CRUD + 合并读取(某语言全量 = 无;某语言覆盖 = translations 行)。
- `kbase/api/routes/i18n.py`:
  - `GET /api/i18n/{lang}` — **公开**(分享页未登录要用),返回该语言 DB 覆盖 `{key:value}`。
  - `GET /api/i18n` — admin,全量(所有 key × 三语当前值,喂管理页)。
  - `PUT /api/i18n` — admin,写覆盖 `{lang,key,value}`(value 空=删除覆盖,回落基线)。
- `kbase/errors.py` — `AppError(code, message, **params)` → 统一转 HTTPException,detail=`{code,params,message}`。业务代码逐步从 `HTTPException(404,"中文")` 迁到 `raise AppError("kb.not_found", "知识库不存在: {id}", id=kb_id)`;未迁的仍返回旧字符串,前端兜底显示,不阻塞。

## 7. 机翻基线(DeepSeek)

`scripts/i18n_machine_translate.py`:读 `zh.json` 扁平化 → 对 en/ms **只翻缺失的 key**(已有译文保留,避免覆盖人工校准)→ 调 DeepSeek(`deepseek-v4-flash`,api.deepseek.com/v1,key 走 env `DEEPSEEK_API_KEY`)批量翻(分批+术语表提示:产品名 KBase/专有名词保持)→ 写回 en.json/ms.json。可重复运行(增量补全)。

## 8. 抽取策略

中文是源,抽取 = 硬编码中文 → `t('key')` + 收进 zh.json。半自动:脚本扫 `.vue` 中文字面量生成候选 key,**人工审 key 命名**。P1 抽 portal(4 组件),P2 抽 admin(~78 文件)。开发期用 vue-i18n-extract/i18n-ally 找漏。

## 9. 测试策略

- **key 一致性**(CI 关键):zh/en/ms 三份 JSON 的 key 集合必须完全一致——写测试扫三文件,缺 key 直接 fail(防漏译/漏 key)。
- 前端:i18n 工厂单测(fallback 到 zh、插值);LanguagePicker 切换+localStorage;错误拦截器 code→t 渲染。
- 后端:translations CRUD/合并 API 测试;AppError → detail 结构测试;`GET /api/i18n/{lang}` 公开可达(未登录)。

## 10. 阶段切片

### P1(端到端能用 + 能改,先交付)
1. vue-i18n 接入:i18n 工厂 + locales 骨架 + languages 清单 + 检测/localStorage + 两 SPA 挂载
2. LanguagePicker 顶栏切换组件(portal + admin)
3. 后端 Translation 表 + i18n API(GET/{lang} 公开 / PUT admin)+ 前端启动合并 DB 覆盖
4. 后端错误 key 化框架(AppError + 前端拦截器)+ 高频错误迁移
5. DeepSeek 机翻脚本
6. 使用端(portal)文案抽取 → zh.json → 机翻 en/ms
7. 翻译管理页基础版(admin 新增「多语言」:key + 三语 + 编辑保存)
8. 测试全绿 + 浏览器实测(切语言/改译文即时生效/报错本地化)

**P1 交付**:使用端中/英/马可切、能问答、报错本地化;后台能改译文即时生效。

### P2(补全 + 增强)
1. 管理端(admin/views/components)全量文案抽取
2. 后端错误 key 化补全(剩余端点)
3. 翻译管理页增强:搜索 / 缺失译文高亮 / 按域筛选 / 一键 DeepSeek 机翻补全新语言 / 导入导出
4. 账号语言偏好(登录用户存账号,跨设备一致)

## 11. 快速接入新语言(设计验证)

1. `scripts/i18n_machine_translate.py --lang xx` 生成 `locales/xx.json`
2. `languages.ts` 加 `{code:'xx', name:'…', flag:'…'}`
3. 完成——DB 覆盖、管理页、切换器自动支持,无结构改动

## 12. 风险

- **两 SPA bundle 隔离**:i18n 各自实例、各拉一次 DB 覆盖——配置用共享工厂避免漂移;languages 清单单一事实源。
- **key 一致性**:靠 CI 测试卡死三语 key 对齐,防止漏译上线。
- **抽取工作量**:82 文件/2.3 万字是大头——P1 只抽 portal 控制首批范围,P2 再全量;抽取脚本半自动 + 人工审 key。
- **机翻质量**:DeepSeek 基线 + 人工校准术语;运营可随时在后台改——这正是"可调整"诉求的兜底。
