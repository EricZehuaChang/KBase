# KBase M5-1 设计文档 — 前端分端改造（使用端 / 管理端分离）

- 日期：2026-07-06
- 状态：授权自主执行（F1 落地中；F2-F4 见实施计划分段占位）
- 前置：M4-2 已合并（standard profile 四适配器 + Docker 双 compose + GCP 100 并发压测，331 backend / 76 frontend）

## 1. 背景与目标

M1-M4 的前端是单一 SPA：一套路由、一套导航，`viewer`/`editor`/`admin` 三种角色共用同一个「侧栏五个 tab」外壳（问答/知识库/检索分析/生成/设置），只靠前端隐藏入口 + 后端 `require_role` 做权限防呆（M4-1 G6）。这在内部演示阶段没问题，但客户侧的真实使用场景是两类完全不同的人：

- **普通用户（多为 viewer）**：只想问答，不关心知识库怎么建、检索参数怎么调——现状的侧栏对他们是噪音，且暴露了一堆点了也会 403 的入口，观感差。
- **知识库管理员（editor/admin）**：需要建库、传文档、跑检索分析、发起生成、管一部分系统设置——这批操作与"问答"在信息架构上天然属于两个不同的心智模型（"用工具"vs"配置工具"）。

M5-1 的目标：把前端拆成**使用端**（`/`，面向全体用户的极简问答）与**管理端**（`/admin`，面向 editor/admin 的知识库管理工作台），两套独立打包、独立信息架构，但共享同一个后端与同一套鉴权模型（后端零改动）。

## 2. 调研结论

对比同类产品（MaxKB、ChatWiki 等国内知识库/Agent 平台）与通用对话产品（LobeChat、OpenWebUI）：

- **双端分离是品类标准做法**：MaxKB、ChatWiki 均把"对话使用"与"知识库管理"分成两个独立入口/前端，管理端功能重、信息密度高（表格/表单/多 tab），使用端极简（会话列表 + 输入框），这不是某家产品的个性化选择，而是知识库类 SaaS 的共识形态——管理复杂度不应该泄漏给最终提问的人。
- **聊天体验的细节值得抄**：LobeChat/OpenWebUI 在"问答"这个单一场景里打磨得更细——内联引用（上标点击展开，而不是现状的抽屉硬跳转感）、快捷问题（空会话时给出建议问句）、停止生成（流式过程中可打断）、会话侧栏的分组与检索。这些属于 F2（使用端问答体验）的输入，本设计文档先记录调研结论，F1 不实现。

结论：M5-1 采用"双端分离 + 使用端体验对齐聊天类产品细节"的方向，分四个任务落地（F1 脚手架 → F2 使用端体验 → F3 管理端信息架构 → F4 视觉 polish 与收尾）。

## 3. 双端架构

### 3.1 使用端 `/`

面向全部角色（viewer/editor/admin 登录后默认落地页）。范围：极简问答——会话侧栏、内联引用上标、快捷问题、停止生成。**F1 只搭脚手架**（顶栏 + 复用现状 ChatView，不做上述体验重构），这些体验项由 F2 落地。

### 3.2 管理端 `/admin`

面向 editor/admin（viewer 无权访问，见 §5）。信息架构从现状"侧栏五个扁平 tab"改为"知识库卡片列表 → 点进 KB 详情页，详情页内部用 tabs 组织（文档管理/命中测试/库设置）"，另有生成、系统设置两个顶级入口。**F1 只搭脚手架**（AdminShell 沿用现状 AppShell 的扁平导航，去掉"问答"tab），信息架构重排由 F3 落地。

### 3.3 隔离原则

两条边界，一条是纵深防御，一条是唯一的强制边界：

- **前端隔离（纵深防御 + 体验）**：Vite 多入口独立打包（`index.html` → 使用端 bundle，`admin.html` → 管理端 bundle），使用端 bundle 的静态导入图不包含任何管理端专属视图（KbView/AnalysisView/GenerateView/SettingsView）。这样即使前端路由判断有疏漏，viewer 用户的浏览器也物理上下载不到管理端代码——降低攻击面、减小首屏体积。这**不是**访问控制机制，只是纵深防御。
- **后端路由级 RBAC（唯一强制边界）**：`kbase/auth/deps.py` 的 `require_role` 依赖对每个 `/api/*` 端点的强制校验（M4-1 G3 落地，零改动）。无论前端怎么分端、怎么隐藏入口，真正拦住越权请求的永远是后端。前端分端做得再彻底，也不能替代这层。

## 4. 登录分流

- **viewer**：只能进使用端。访问 `/admin` 时（无论是否已登录）落地到一个不循环跳转的"无权限"提示页（见 F1 落地细则），提供返回使用端的链接。
- **editor/admin**：两端都能进。使用端顶栏有"进入工作台"入口（仅这两个角色可见），管理端顶栏有"返回问答"入口。两个入口都是**整页跳转**（`location.href`），不是前端路由 push——因为两端是完全独立的 bundle，没有共享的 router 实例可以跨 bundle 导航。
- 登录页本身按入口分流：从 `/login` 登录留在使用端，从 `/admin/login` 登录留在管理端（各自 `redirectTarget` 复用现有的开放重定向防护逻辑，见 `src/lib/auth-utils.ts`）。

## 5. 非目标

- **分享链接 / iframe 嵌入**：M6 范围，M5-1 不做。
- **视觉主题重做**：沿用 M2 确立的 design tokens（`src/styles/tokens.css`），不引入新配色/新组件库，两端视觉一致性问题留给 F4 走查处理。
- **后端权限模型改动**：`require_role`/角色矩阵/审计/许可证等 M4-1 落地的鉴权全家桶零改动——M5-1 纯粹是前端信息架构与打包方式调整，后端只多一处静态文件路由分流逻辑（`/admin` 前缀回退到 `admin.html`）。

## 6. 风险与预案

- **Vue Teleport 跨 bundle 失效**：现状 `ChatView.vue` 用 `Teleport to="#sidebar-slot"` 把会话列表注入 `AppShell` 提供的挂载点。F1 中 `PortalShell.vue` 复用 `ChatView` 但不做重构，因此必须在 `PortalShell` 里保留同名挂载点，否则 Teleport 目标缺失会静默丢失会话侧栏（Vue 只 warning，不报错，容易被忽略）——F1 验收时需手动核实该挂载点存在且非空渲染。F2 重构 `ChatView` 时会连带清理这层耦合。
- **manifest 结构随 Vite 版本变化**：`check-bundle-isolation.mjs` 依赖 `build.manifest` 产物的字段结构（`imports`/`dynamicImports`）与文件路径（`.vite/manifest.json` vs `manifest.json`），不同大版本可能有差异——脚本按实际构建产物落地校验，不臆测字段名，锁定当前 Vite 版本行为，后续升级 Vite 需重新核对。
- **两套 `main.ts` 各自初始化全局状态**：`theme.ts`（FOUC 预绘制）、`currentRole`（会话角色响应式变量）等模块级单例状态在两个 bundle 里是**各自独立的实例**（不同 JS 上下文），不会共享——这是预期行为（两个 tab 打开使用端和管理端本来就该是独立会话视图），但要避免误以为"改一个全局状态两端会同步"。
