# KBase M4-1（商业化加固）实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 认证（JWT Cookie + API Key 双通道）、三级角色、审计日志、API Key 管理、轻量许可证；前端登录与管理界面。

**Architecture:** 见 spec `docs/superpowers/specs/2026-07-06-kbase-m4-1-design.md`。新模块 `kbase/auth/`（security.py 哈希与 JWT / deps.py FastAPI 依赖与角色映射 / bootstrap.py 首启 admin）、`kbase/audit.py`、`kbase/license.py`。测试口径 `create_app(auth="on"|"off")`（spec §7）。

**Tech Stack:** + passlib[bcrypt]、PyJWT、cryptography(Ed25519,随 mcp 依赖树可能已有——确认)。

**基线：** main @ 6492d4e（161 backend / 55 frontend）。分支 `feature/m4-1`。约定沿前例。

---

### Task G1: 数据层与安全原语
users/api_keys/audit_logs 三表（models+create_all）；kbase/auth/security.py（bcrypt hash/verify、JWT encode/decode(exp 7d)、secret 解析：env 否则 app_settings 生成持久化、api key 生成 `kbase_ak_`+32url 随机与 sha256 校验）；kbase/auth/bootstrap.py（ensure_admin：空表建 admin，env 密码否则随机打日志）。TDD：哈希往返/坏密码、JWT 往返/过期/篡改、secret 持久化稳定、key 生成校验、bootstrap 幂等与 env 优先。deps 加 passlib[bcrypt]+PyJWT。
Commit `feat: 认证数据层与安全原语（bcrypt/JWT/APIKey/bootstrap）`

### Task G2: 会话端点与全局鉴权
kbase/auth/deps.py：`get_current_actor` 依赖（Cookie JWT → user；Bearer kbase_ak_ → key 记录；均无 → 401）；`require_role(min_role)` 工厂（admin>editor>viewer 序）；Origin 同源中间件（非 GET 且带 Origin 时校验，不匹配 403）。api/main.py：auth 路由（login 校验+签发 Cookie/logout 清除/me）；`create_app(auth="on")` 参数——on 时对全部 /api 路由挂 actor 依赖（router-level dependency），豁免清单 per spec §2；既有全部功能测试改传 auth="off"（机械替换 _client 辅助）；bootstrap 在 create_app 调用。新测试 tests/test_auth.py：登录成/败/禁用户、无凭据 401、Cookie 会话贯通、logout 后 401、Origin 不匹配 403、豁免路径可达。
Commit `feat: 登录会话与全局鉴权（Cookie/Bearer 双通道+Origin 防护）`

### Task G3: 角色矩阵与审计
路由→最低角色映射（spec §3 表落成代码：viewer 集合只读+问答；editor 加内容管理；admin 全量——以依赖注入实现在各路由）；kbase/audit.py：write_audit(sf, actor, action, resource, detail, ip) + 中间件（mutating 全记、login 成败、query 记问题前100字）；GET /api/audit（admin，分页 {items,total}）。测试：角色矩阵抽查（viewer 传/删 403、editor 设置 403、admin 全通）、审计行断言（登录失败也落行）、audit API 分页与权限。
Commit `feat: 三级角色矩阵与审计日志`

### Task G4: API Key 管理、MCP 接入与许可证
设置 API：POST /api/settings/api-keys（admin，返回完整 key 一次）/GET 列表(prefix+name+role+revoked)/DELETE 吊销；Bearer 通道接 api_keys 校验（revoked 401）。kbase_mcp：env KBASE_API_KEY → 所有请求 Authorization 头；未配置时工具 error 指引。kbase/license.py：Ed25519 验签（公钥常量）、状态枚举 trial/valid/expired/invalid、GET /api/license；scripts/gen_license.py（生成密钥对+签发，README 说明私钥保管）。测试：key 全生命周期（建→用→吊销→401）、MCP 工具带 key 过 ASGI 鉴权 on 应用、license 四态。
Commit `feat: API Key 管理+MCP 鉴权接入+轻量许可证`

### Task G5: 前端登录与会话
LoginView（用户名/密码/错误提示，登录后跳原目标）；router 守卫（/login 外全部需会话——用 GET /api/auth/me 探测，模块级缓存）；api.ts 401 拦截→清缓存跳 /login；AppShell 底部用户块（用户名/角色徽章/登出）。vitest：守卫与 401 拦截逻辑纯函数化测试。所有页面在未登录时不可达（手动 dev 验证）。
Commit `feat(web): 登录页与会话守卫（401 拦截+用户菜单）`

### Task G6: 前端管理界面与角色门控
设置页新增（admin 可见）：用户管理卡片（列表/新建[用户名+角色+初始密码]/重置密码/禁用启用）+ API Key 卡片（列表 prefix/name/role、新建弹窗一次性展示、吊销确认）+ 许可证状态卡片与全局横幅（trial/expired/invalid 时 AppShell 顶部细条横幅）。角色门控：me.role 注入全局状态；viewer 隐藏上传/删除/新建库/生成发起/设置入口；editor 隐藏设置入口。后端补用户管理 API（admin：GET/POST /api/users、PUT /api/users/{id}(重置密码/禁用/角色)，含"不能禁用最后一个 admin"防护+测试）。vitest：门控纯函数。
Commit `feat: 用户管理与 API Key 界面+角色门控+许可证横幅`

### Task G7: 真实验收与文档
起真实服务（auth on）：bootstrap admin 日志密码登录 → 建 editor/viewer 用户 → 三角色浏览器/脚本矩阵抽查（viewer 删除按钮不可见且 API 403）→ 建 API Key → MCP 带 key 实调三工具 → 审计页有登录/查询/变更记录 → license 试用横幅显示。README「安全与部署」章节（首启密码、角色说明、API Key、license 签发、KBASE_SECRET_KEY 建议显式设置）。全量回归（前后端）。
Commit `docs: README 安全章节`
