# KBase M5-1（前端分端改造）实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 使用端（`/`）与管理端（`/admin`）拆成两个独立 Vite bundle，登录按角色分流，后端双 SPA 服务；在此脚手架上分两步做体验/信息架构重排，最后视觉走查收尾。

**Architecture:** 见 spec `docs/superpowers/specs/2026-07-06-kbase-m5-1-design.md`。前端隔离靠 Vite 多入口独立打包，后端 RBAC（`require_role`）零改动、仍是唯一强制边界。

**Tech Stack:** 无新依赖，纯前端构建配置 + 路由改造；后端仅 `SPAStaticFiles` 分流逻辑扩展。

**基线：** main @ a1e69df（331 backend / 76 frontend）。分支 `feature/m5-1`。约定沿前例。

---

### Task F1: Vite 多入口 + 双 Shell + 登录分流 + 后端双 SPA 服务

**目标**：把现状单一 SPA（`src/main.ts` → `App.vue` → `AppShell.vue` 一套路由五个 tab）拆成两个独立入口——使用端 `index.html`（`src/portal/`）与管理端 `admin.html`（`src/admin/`）——各自独立打包、独立路由、独立登录分流，后端按路径前缀分流回退到对应的 `index.html`/`admin.html`。

1. **Vite 多入口**（`web-app/vite.config.ts`）：`build.rollupOptions.input = {main: 'index.html', admin: 'admin.html'}`，开启 `build.manifest: true`（供隔离校验脚本解析产物依赖图）。新建 `web-app/admin.html`（挂载 `src/admin/main.ts`，独立 `<title>` 便于后端序列测试用 marker 区分）。`src/` 重组：
   - `src/portal/`：`main.ts`、`router.ts`（`createWebHistory('/')`）、`PortalShell.vue`——极简：顶栏（logo + 用户块 + "进入工作台"，仅 `canManageContent` 角色可见）+ `ChatView`（Teleport 挂载点必须保留，见 spec 风险条目）。
   - `src/admin/`：`main.ts`、`router.ts`（`createWebHistory('/admin')`）、`AdminShell.vue`（≈ 现状 `AppShell.vue`，导航去掉"问答"）、`guard.ts`（纯函数，角色落地判定）、`ForbiddenView.vue`（无权限提示页）。
   - 共享不动：`src/lib`、`src/components`（除 `AppShell.vue` 被 `AdminShell.vue` 取代而删除）、`src/composables`、`src/styles`。
   - `src/views/`：`ChatView.vue`/`LoginView.vue` 两端共用（原样保留，物理位置不变）；`KbView.vue`/`AnalysisView.vue`/`GenerateView.vue`/`SettingsView.vue` 只在 `src/admin/router.ts` 里以**路由级懒加载**（`() => import(...)`）接线——这既是信息架构上的"仅管理端路由可达"，也是产生独立 chunk 供隔离校验脚本断言的前提（静态 import 不会切出独立 chunk，检查会因为"manifest 里压根找不到该模块"而失去意义）。
   - 删除：`src/main.ts`、`src/router.ts`、`src/App.vue`、`src/components/AppShell.vue`（内容迁移进 `src/portal`/`src/admin` 对应文件）。

2. **Bundle 隔离（本任务的核心验收）**：新增 `web-app/scripts/check-bundle-isolation.mjs`——构建后读取 `build.manifest` 产物，从 `index.html`（使用端入口）出发做可达性遍历（含 `imports` 与 `dynamicImports`），断言遍历结果里不包含任何 `src/admin/*` 模块或 `KbView`/`AnalysisView`/`GenerateView`/`SettingsView` 四个管理端专属视图；另做一次反向检查——确认这些模块**确实存在**于 manifest 中（防止"因为压根没接线导致的假阳性通过"）。挂 `package.json` script `check-isolation`，验证阶段跑一次并把关键证据（reachable 模块数、被排除的管理端模块列表）记进报告。

3. **角色落地与守卫**（`src/admin/guard.ts` + `src/admin/router.ts` 的 `beforeEach`）：
   - 无会话 → 跳 `/admin/login`（带 `redirect`，复用 `auth-utils.ts` 的 `loginRedirectQuery`）。
   - 有会话且 `role === "viewer"` → 跳 `/admin/forbidden`（终态路由，自身不再被守卫拦截，避免循环重定向）。
   - 有会话且 `role` 为 `editor`/`admin` → 放行，`/admin` 根路径渲染知识库页（`KbView`）作为管理端首页。
   - `PortalShell` 的"进入工作台"按钮（`canManageContent(role)` 才可见）与 `AdminShell` 的"返回问答"按钮都用 `location.href` 整页跳转（两端是独立 bundle，没有共享 router 实例可以 `router.push` 跨过去）。
   - vitest 覆盖：`guard.ts` 的落地判定纯函数（无会话/viewer/editor/admin 三×几种目标路径的矩阵）+ 沿用现状 `auth-utils.test.ts` 风格。

4. **后端双 SPA 服务**（`kbase/api/main.py` `SPAStaticFiles`）：路径以 `/admin` 开头且不是真实静态文件、不是 `/api/*`、不在 docs 排除清单里 → 404 时回退 `admin.html`；其余非文件 404 → 回退 `index.html`（现状行为不变）；`/openapi.json`/`/docs`/`/redoc` 在 `auth="on"` 下继续 404、静态资源鉴权豁免行为不变。新增后端测试（`auth="on"`）：`GET /admin` → 200 且命中 `admin.html` marker；`GET /` → 200 且命中 `index.html` marker；`GET /admin/some-deep-route` → 200 回退 `admin.html`（SPA 深链接）；`/api/kb` 无凭据 → 401；`/openapi.json` → 404（不受本次改动影响，回归确认）。

5. **回归**：后端从 worktree 跑全量（`pytest --collect-only -q` 先确认收集自本 worktree），如实报告新基线（预期 331 + 本任务新增的 SPA 分流测试数）；前端 `npm ci && npm test` 绿（76 基线 + 本任务新增 guard 测试），`npm run build` 干净，`npm run check-isolation` 通过；构建产物 `web/` 按仓库既有约定提交（`web-app/` 是源码、`web/` 是构建产物，两者都入库）。

6. **提交**：先提交本文档 + spec（中文 message），再提交实现：`feat(web): Vite 多入口分端脚手架（独立 bundle+登录分流+双 SPA 服务）`。完成后 `git push -u origin feature/m5-1`。

### Task F2: 使用端问答体验（占位）

在 F1 的 `PortalShell`/`ChatView` 脚手架之上，落地 spec §3.1 列出的体验项：会话侧栏重做（脱离对 `AppShell` `#sidebar-slot` Teleport 挂载点的耦合，`PortalShell` 原生持有会话状态）、内联引用上标（替换/补充现状 `CitationDrawer` 的硬跳转体验）、空会话快捷问题建议、流式生成过程中的停止按钮。具体拆分与验收标准在 F1 合并后另开设计小节或直接细化于本文件（视 F1 落地时暴露的耦合点决定颗粒度）。

### Task F3: 管理端信息架构（占位）

在 F1 的 `AdminShell` 扁平导航之上，按 spec §3.2 重排：知识库卡片列表首页 → 点进 KB 详情页，详情内部用 tabs 组织（文档管理/命中测试/库设置，对应现状 `KbView`/`AnalysisView` 里按 KB 维度的功能重新分组），生成、系统设置维持顶级入口。具体拆分待 F1/F2 落地后明确。

### Task F4: 视觉 polish + 浏览器走查 + 合并（占位）

两端视觉一致性检查（沿用 M2 tokens，不新增配色）、真实浏览器手动走查三角色×两端矩阵、全量回归、终审合并 `feature/m5-1` → `main`。
